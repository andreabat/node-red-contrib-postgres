<decisions>
## Locked Decisions (MUST implement exactly)

- **D-04:** Auto-generated prepared statement names from query hash (e.g., `ps_abc123`). No user-facing UI for naming. Every unique query automatically becomes a prepared statement. Transparent to user.
- **D-05:** Per-connection cache — each pool client tracks its own prepared statements. Statements survive across multiple queries on the same client connection.
- **D-06:** Redeploy clears cache — all prepared statements for a node are invalidated on node redeploy. Queries re-prepare on first execution post-deploy. Pool recreation (new pg.Pool from PostgresDBNode) naturally clears all prepared statements.

## the agent's Discretion (resolved in this plan)

- **SQL editor (QUERY-04):** Use ACE editor (bundled with Node-RED 3.x, already in use) — NOT CodeMirror. RESEARCH.md confirms ACE is bundled and CodeMirror would add unnecessary npm dependency. Enhancements: `showLineNumbers: true`, `wrap: true`, `useWorker: false`, `maxLines: Infinity`, `fontSize: 14`, `enableBasicAutocompletion: true`, `enableLiveAutocompletion: true`. Theme: `tomorrow` (matches Node-RED editor).
- **Type mapping behavior (REL-03):** Opt-in per node, off by default (no breaking change). Three separate checkboxes: "Map NUMERIC → Number" (`mapNumeric`), "Map TIMESTAMPTZ → ISO 8601" (`mapTimestamptz`), "Auto-parse JSONB" (`parseJsonb`). JSONB is already parsed by pg by default — the toggle controls whether to use pg's default parser (on) or raw string (off). Type parsers registered globally via `pg.types.setTypeParser` at pool creation; per-query opt-out via `types` config when all toggles are off.
- **Prepared statement hashing:** Use MD5 hash truncated to 8 hex chars with `ps_` prefix (max 11 chars, well within PostgreSQL's 63-byte identifier limit). RESEARCH.md anti-pattern #6 warns against SHA-256 (64 chars exceeds limit).
- **JSONB toggle behavior:** pg already parses JSONB by default (OID 3807). When `parseJsonb` is ON (default per-node), use pg's built-in parser. When OFF, override with identity parser returning raw string.
</decisions>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| ACE editor (browser) → hidden input | SQL query text crosses from editor to DOM — standard Node-RED editor pattern, no new risk |
| pg.types.setTypeParser | Global type parser registration affects all pools in the Node-RED process — RESEARCH.md assumption A4 flags this as a cross-contrib-node risk |
| Prepared statement name hash | MD5-truncated hash used as SQL identifier — collision risk is theoretical (~1 in 4B per node's query set) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-09 | Tampering | Type mapping precision loss | mitigate | `mapNumeric` off by default (opt-in); tooltip warns about precision loss for values > 2^53; `mapTimestamptz` off by default; JSONB on by default matches pg's existing behavior (not a change) |
| T-02-10 | Tampering | Global pg.types.setTypeParser side effect | accept | Type parsers registered globally affect all pg-using code in the Node-RED process (RESEARCH.md assumption A4). Risk is low: NUMERIC→parseFloat and INT8→parseInt are reasonable defaults; TIMESTAMPTZ→ISO string is a behavioral change. Per-node opt-out via `types` config allows bypassing global parsers when needed. |
| T-02-11 | Tampering | Prepared statement name collision | accept | MD5 truncated to 8 hex chars (~4 billion namespace). Collision within a single node's query set is extremely unlikely. Even if collision occurs, pg would return "prepared statement already exists" error, caught by the existing error handler — not a silent data corruption risk. |
| T-02-12 | Information Disclosure | ACE editor autocomplete | accept | ACE editor `enableBasicAutocompletion` and `enableLiveAutocompletion` use local word completion — no network requests, no data exfiltration risk |
</threat_model>
