<objective>
Add atomic multi-step transaction support to PostgresNode via a config toggle. When `transactionMode` is ON and `msg.payload` is an array of `{query, params, output}` entries, the node acquires a single pool client and executes all queries within BEGIN/COMMIT/ROLLBACK. On any failure, auto-ROLLBACK with structured error; client release guaranteed in finally.

Purpose: Enable Node-RED developers to execute dependent SQL statements atomically without a dedicated transaction node type, using the familiar PostgresNode config panel.
Output: PostgresNode with transaction mode toggle, TransactionQuery type, transaction test suite, i18n strings
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| msg.payload from upstream Node-RED nodes | Untrusted user data enters as transaction query array |
| Mustache-rendered query strings | msg fields interpolated into SQL before execution |
| PostgreSQL return values | Database responses placed on msg.payload |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | Mustache-rendered SQL in transaction entry queries | mitigate | Each transaction entry.query is Mustache-rendered, then values are passed via parameterized queries ($1, $2). Same Phase 2 protection — Mustache templates coexist with parameterized queries; bindNamedParams runs after Mustache. |
| T-03-02 | Denial of Service | Transaction client not released on error | mitigate | `client.release()` in `finally` block (line 94-113 of PostgresNode.ts). Covers ALL paths: success, query failure, ROLLBACK failure, timeout. Release is wrapped in inner try/catch to prevent release errors from blocking cleanup. |
| T-03-03 | Denial of Service | Pool exhaustion from concurrent long-running transactions | accept | Transactions use a single pool client — same pool as single queries. Pool max is configured by developer (POOL-01). If a transaction holds a client for a long time, it reduces available pool connections — but this is expected behavior, controlled by the developer's query design. No worse than a slow single query. |
| T-03-04 | Repudiation | No audit trail for transaction success/failure | accept | Node-RED's built-in logging captures node.error() calls. Transaction ROLLBACK errors are logged via node.warn(). No additional audit trail needed for v1 — this is a contrib node, not a system of record. |
</threat_model>
