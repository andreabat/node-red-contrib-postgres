<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Node-RED editor → Config node | User-supplied connection params (host, port, database, pool settings) cross from untrusted UI to runtime config node |
| Node-RED flow msg → Query node | User-supplied msg payload and msg.params cross from untrusted flow to SQL execution |
| Listener channel → NOTIFY handler | PostgreSQL NOTIFY payloads cross from external DB into Node-RED runtime |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering | getField.ts num case | mitigate | `Number()` + `isNaN()` guard with `node.warn()` — invalid numeric inputs are caught and warned instead of silently producing NaN (addressed in Plan 01 migration) |
| T-01-02 | Tampering | getField.ts bool case | mitigate | try/catch around `JSON.parse` with fallback to `value === 'true'` string comparison — malformed bool inputs do not throw (addressed in Plan 01 migration) |
| T-01-03 | Information Disclosure | PostgresDBNode credentials | mitigate | User/password stored in Node-RED credential storage (not plain config); credential schema `{ user: { type: 'text' }, password: { type: 'password' } }` ensures password is masked in editor UI — no change from pre-migration |
| T-01-04 | Spoofing | Mustache SQL templates | accept | Mustache renders msg fields directly into SQL strings — this is a known SQL injection risk inherent to template-based query building. Mitigation (parameterized queries, input sanitization) is deferred to Phase 2 (QUERY-01, QUERY-02). Pre-migration behavior is preserved exactly. |
| T-01-05 | Injection | LISTEN channel name | accept | Channel name is interpolated directly into `LISTEN ${config.channel}` — no sanitization. Mitigation (pg-format identifier escaping) is deferred to Phase 3 (LISTEN-02). Pre-migration behavior preserved. |
| T-01-06 | Information Disclosure | console.log calls | mitigate | Replaced with `node.debug()` / `node.log()` — debug output is filtered by Node-RED log level, not unconditionally written to stdout (BUG-03, addressed in Plan 01-02) |
| T-01-07 | Elevation of Privilege | TypeScript strict mode | mitigate | `strict: true` in tsconfig.json catches implicit any, null reference, and type mismatch errors at compile time — preventing a class of runtime bugs in module-level code |

**No blocking threats:** All `mitigate` threats are addressed in this phase. `accept` threats (T-01-04, T-01-05) are documented for future phases and represent preserved pre-migration behavior.
</threat_model>
