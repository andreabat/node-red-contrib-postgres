<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Node-RED flow msg → Query node | User-supplied msg payload and msg.params cross from untrusted flow to SQL execution |
| Mustache rendering | msg properties interpolated directly into SQL query strings |
| pg.Pool client release | Connection lifecycle must guarantee release even on error paths |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-04 | Spoofing | Mustache SQL templates | accept (documented) | Mustache renders msg fields directly into SQL strings — SQL injection vector. Mitigation deferred to Phase 2 (QUERY-01 named parameters). Test 4 verifies rendering behavior is preserved exactly. |
| T-01-06 | Information Disclosure | PostgresNode query logging | mitigate | `console.log` replaced with `node.debug()`/`node.log()` — query content logged at debug level (filtered by Node-RED log config). Addressed in this task. |
| T-01-08 | Denial of Service | Client release guarantee | mitigate | `client.release()` is called in `finally` block with nested try/catch — ensures release even on query failure or release error. Verified by Tests 1-3. |
| T-01-09 | Information Disclosure | Error messages in msg | accept (preserved behavior) | `msg.error` string includes raw PostgreSQL error messages when `throwErrors=false`. This is pre-migration behavior preserved exactly. Structured errors (QUERY-02) deferred to Phase 2. |
</threat_model>
